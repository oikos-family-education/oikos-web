# 10. Internationalisation (i18n)

Canonical rules: [.claude/rules/i18n-guidelines.md](../.claude/rules/i18n-guidelines.md).

## Stack

- **`next-intl` v4** with static message bundles.
- Single locale shipped today: **English (`en`)**.
- Messages live in [apps/web/messages/en.json](../apps/web/messages/en.json).
- Every URL is locale-prefixed via `app/[locale]/…`.

## Golden rule

**No hardcoded user-facing strings in components.** Every label, placeholder, button, toast, and error message must flow through `useTranslations`.

## Namespace conventions

- Namespaces are **PascalCase**: `Auth`, `Dashboard`, `Onboarding`, `Planner`, `Projects`, …
- Keys inside a namespace are **camelCase**: `signInButton`, `emailRequired`.
- **Flat structure only** — namespace → key. No nested objects within a namespace.
- One namespace per feature or page.

Existing namespaces include: `Home`, `Auth`, `Validation`, `ApiErrors`, `PasswordStrength`, `Onboarding`, `Dashboard`, `Navigation`, `Cards`, `Placeholder`, plus feature-scoped ones as the app grows.

## Usage patterns

```tsx
const t = useTranslations('Dashboard');

<h1>{t('title')}</h1>
<p>{t('welcome', { name: user.first_name })}</p>

// Rich content:
<p>
  {t.rich('agreeToTerms', {
    link: (chunks) => <a href="/terms">{chunks}</a>,
  })}
</p>
```

## Zod + translations

Validation schemas that use translated messages must be memoised, because `useTranslations` returns a new function every render:

```tsx
const tVal = useTranslations('Validation');

const schema = useMemo(() =>
  z.object({
    email: z.string().min(1, tVal('emailRequired')).email(tVal('emailInvalid')),
    password: z.string().min(10, tVal('passwordTooShort')),
  }),
[tVal]);
```

## Adding a new locale

1. Add `apps/web/messages/<locale>.json` with the same keys as `en.json`.
2. Register the locale in [apps/web/i18n.ts](../apps/web/i18n.ts).
3. Update `middleware.ts` so the new locale is recognised.
4. (Optional) Add a language switcher — we don't ship one yet.

## Adding a new namespace or key

1. Edit `messages/en.json` — keep it sorted alphabetically by key within the namespace if possible.
2. Call `useTranslations('YourNamespace')` in the component.
3. Run the app; `next-intl` will console-warn for any missing keys.

The `/oikos-i18n` skill automates most of this.

## Common mistakes

- Using `t(someVariable)` — keys must be string literals for static analysis. Build a switch if needed.
- Forgetting to memoise the Zod schema — each render creates a new schema and RHF loses focus on inputs.
- Splitting one feature across two namespaces "because it felt cleaner". Keep one namespace per feature.
